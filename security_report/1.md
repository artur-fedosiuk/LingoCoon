## 1. Authentication & Session Management

### 1.1 — Session Refresh via Middleware
- **Status**: ✅ Secure
- **Why**: `src/middleware.ts` correctly calls `supabase.auth.getUser()` (the server-validated JWT check, not the local cache) on every request and refreshes session cookies in the response. This follows Supabase SSR best practices exactly.
- **Attack scenario**: N/A
- **Fix**: No action needed.

### 1.2 — Open Redirect in OAuth Callback
- **Status**: 🔴 At Risk
- **Why**: `src/app/auth/callback/route.ts` line `11` reads the `next` query parameter directly from the URL and then redirects to it with no validation:
  ```ts
  const next = searchParams.get('next') ?? '/dashboard'
  // …
  return NextResponse.redirect(`${origin}${next}`)
  ```
  An open redirect (redirecting a user to an attacker-controlled URL) can be used in phishing — the victim clicks a link that starts on your trusted domain and ends up on a fake site.
- **Attack scenario**: Attacker sends victim the link `https://lingocoon.app/auth/callback?code=VALID_OAUTH_CODE&next=//evil.com/steal`. After the OAuth flow completes, the browser is silently redirected to `evil.com`. Because the link looked like it came from LingoCoon, the victim trusts it.
- **Fix**: Validate that `next` is a relative path only:
  ```ts
  const rawNext = searchParams.get('next') ?? '/dashboard';
  // Only allow relative paths starting with /
  const next = rawNext.startsWith('/') && !rawNext.startsWith('//') ? rawNext : '/dashboard';
  ```

### 1.3 — Signup Route Not Properly Guarded
- **Status**: 🟡 Unknown
- **Why**: The middleware marks `/signup*` as a public route (line `47` of `middleware.ts`), but no `src/app/signup` directory exists in the project. This is a dead entry in the allowlist — low risk now, but if a `/signup` page is added later, it won't get redirected for already-authenticated users, which is confusing UX.
- **Fix**: Remove the `/signup` entry from the public routes list if no signup page exists, or add the missing redirect guard when it is created.

## 2. Database Security

### 2.1 — `getDeck()` Has No User Authorization Check
- **Status**: 🔴 **Critical**
- **Why**: `src/lib/actions/deck-actions.ts` lines `101–116` fetch a deck by ID without verifying the requester owns it:
  ```ts
  export async function getDeck(deckId: string) {
    const supabase = await createClient();
    // ❌ No supabase.auth.getUser() call here
    const { data, error } = await supabase
      .from('decks')
      .select('*')
      .eq('id', deckId)  // ← any ID works, no .eq('user_id', user.id)
      .single();
    return { deck: data };
  }
  ```
  This is an IDOR (Insecure Direct Object Reference) — a bug where knowing an object's ID is enough to access it, regardless of who owns it. Deck IDs are UUIDs, but UUIDs are not secret.
- **Attack scenario**: Once logged in, an attacker calls `getDeck('some-other-users-deck-uuid')`. If RLS is disabled or misconfigured on the `decks` table, they get the full deck contents back. UUIDs are often embedded in URLs, so any user who shared a link has leaked their deck ID.
- **Fix**: Add authentication and ownership check:
  ```ts
  export async function getDeck(deckId: string) {
    const supabase = await createClient();
    const { data: { user } } = await supabase.auth.getUser();
    if (!user) return { error: 'Not authenticated' };
    const { data, error } = await supabase
      .from('decks')
      .select('*')
      .eq('id', deckId)
      .eq('user_id', user.id)  // ← add this
      .single();
    return error || !data ? { error: 'Not found' } : { deck: data };
  }
  ```

### 2.2 — `getCards()` Has No User Authorization Check
- **Status**: 🔴 **Critical**
- **Why**: `src/lib/actions/deck-actions.ts` lines `144–159` also skip the auth check entirely. Any authenticated user can call `getCards('any-deck-id')` and read all cards in that deck — which may include personal language study notes or sensitive content.
- **Attack scenario**: Same as 2.1 — if an attacker discovers a `deckId` (e.g., from a shared URL or network tab), they can enumerate all cards without owning the deck.
- **Fix**: Same pattern — call `supabase.auth.getUser()`, then either add `.eq('deck_id', deckId)` + verify deck ownership first, or rely on tight RLS. Do not do one without the other.

### 2.3 — Row Level Security (RLS) Status Unknown
- **Status**: 🟡 Unknown
- **Why**: The application code does not include any Supabase migration files or SQL schema in the repo, so it's impossible to verify RLS policies from the source code alone.

  RLS (Row Level Security) is a database feature that restricts which rows each user can see or modify, enforced at the database level. Even if your server code has bugs like 2.1 and 2.2, correct RLS stops the leak.
- **Attack scenario**: If RLS is disabled on `decks` or `cards`, the bugs in 2.1 and 2.2 are immediately exploitable. If RLS is enabled, they are partially mitigated — but relying on RLS alone without server-side auth checks is a single point of failure.
- **Fix**: In the Supabase dashboard, go to **Table Editor** → your table → **RLS**. Ensure policies like:
  ```sql
  -- decks: users can only see their own
  CREATE POLICY "Owner only" ON decks
    FOR ALL USING (auth.uid() = user_id);
  ```
  are active. Then keep the server-side auth checks in the code as defense-in-depth.

### 2.4 — SQL Injection Risk
- **Status**: ✅ Secure
- **Why**: All database queries use the Supabase JS client's typed query builder (`.from('decks').select('*').eq('id', deckId)`), which uses parameterized queries internally. No raw SQL string interpolation or `.rpc()` calls with unsanitized input were found. SQL injection (inserting malicious SQL through user input) is not feasible here.
- **Fix**: No action needed. Keep using the typed builder.

## 3. API & Server-Side

### 3.1 — TTS API Routes Have No Authentication
- **Status**: 🔴 **At Risk**
- **Why**: Both `/api/tts/synthesize` (`src/app/api/tts/synthesize/route.ts`) and `/api/tts/voices` (`src/app/api/tts/voices/route.ts`) are completely open — there is no `supabase.auth.getUser()` call, no session check, nothing. The middleware explicitly skips all `/api/*` routes (line `49` of `middleware.ts`):
  ```ts
  request.nextUrl.pathname.startsWith('/api/'); // Allow API routes (they handle their own auth)
  ```
  But these routes don't handle their own auth.
- **Attack scenario**: Anyone on the internet (no login required) can POST to `https://lingocoon.app/api/tts/synthesize` with arbitrary text and make it talk, burning your Google Cloud TTS quota. With the free tier, this could suspend your billing account. With a paid account, an attacker could rack up costs in minutes.
- **Fix**: Add an auth check at the top of both API route handlers:
  ```ts
  import { createClient } from '@/lib/supabase/server';
  export async function POST(request: NextRequest) {
    const supabase = await createClient();
    const { data: { user } } = await supabase.auth.getUser();
    if (!user) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
    // … rest of handler
  }
  ```

### 3.2 — Internal Error Details Leaked in API Response
- **Status**: 🟡 At Risk
- **Why**: `src/app/api/tts/synthesize/route.ts` line `171` returns the raw JavaScript error as a string:
  ```ts
  return NextResponse.json({ error: 'Internal server error', detail: String(error) }, { status: 500 });
  ```
  This can expose internal paths, stack traces, or library error messages to the browser. Information leakage (giving attackers clues about your internals) is a real attack enabler.
- **Fix**: Log the raw error server-side only, return a generic message to the client:
  ```ts
  console.error('TTS API route error:', error);  // keep this
  return NextResponse.json({ error: 'Internal server error' }, { status: 500 }); // remove 'detail'
  ```

### 3.3 — Input Validation: No Schema on TTS Request Body
- **Status**: 🟡 At Risk
- **Why**: The TTS synthesize route casts the body directly as `TTSRequest` (line `100`) without schema validation. While TypeScript types are stripped at runtime, numeric fields like `pitch`, `speakingRate`, and `volumeGainDb` are passed directly to the Google API without checking type or range. No Zod schema is used anywhere in the API routes.
- **Attack scenario**: A caller sends `{"text": "hello", "languageCode": "en-US", "pitch": 99999}`. This gets forwarded to Google TTS. The behavior is Google's problem to handle, but you lose the ability to enforce your own constraints and may get unexpected billing or API errors that leak information.
- **Fix**: Add a Zod schema (Zod is a TypeScript-friendly library for parsing and validating data at runtime):
  ```ts
  import { z } from 'zod';
  const TTSSchema = z.object({
    text: z.string().min(1).max(5000),
    languageCode: z.string().regex(/^[a-z]{2}-[A-Z]{2}$/),
    pitch: z.number().min(-20).max(20).optional(),
    speakingRate: z.number().min(0.25).max(4.0).optional(),
  });
  const body = TTSSchema.safeParse(await request.json());
  if (!body.success) return NextResponse.json({ error: 'Invalid input' }, { status: 400 });
  ```

### 3.4 — `deleteCardAction` in Old File Has No Ownership Check
- **Status**: 🔴 **At Risk**
- **Why**: `src/lib/actions/deck-actions.ts` line `190–211` (the older duplicate action file) deletes a card by ID with only an auth check, no ownership verification:
  ```ts
  const { error } = await supabase
    .from('cards')
    .delete()
    .eq('id', cardId); // ← no check that this card belongs to user
  ```
  There are two `deck-actions.ts` files in this project (one under `src/app/actions/`, another under `src/lib/actions/`). The newer one in `src/app/actions/` does check ownership. The older one is dangerous if still called anywhere.
- **Fix**: Audit all callers of `src/lib/actions/deck-actions.ts` and redirect them to the safe version in `src/app/actions/deck-actions.ts`. Then delete the old file.

## 4. Frontend

### 4.1 — XSS Risk
- **Status**: ✅ Secure
- **Why**: No `dangerouslySetInnerHTML` was found anywhere in the component tree. React's JSX escapes all string values by default. No usage of `eval()`, `innerHTML`, or `document.write` detected.
- **Fix**: No action needed. Keep avoiding `dangerouslySetInnerHTML`.

### 4.2 — Client Bundle Exposure
- **Status**: ✅ Secure
- **Why**: The only `NEXT_PUBLIC_` variables are `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY`, both of which are intentionally public — client-side Supabase requires them. No service role key, no Google private key, and no Google OAuth client secret use the `NEXT_PUBLIC_` prefix.
- **Fix**: No action needed. Maintain this discipline. **Important**: never add `NEXT_PUBLIC_` prefix to `SUPABASE_AUTH_GOOGLE_CLIENT_SECRET` or `GOOGLE_TTS_API_KEY`.

### 4.3 — Translation Service Calls Made Directly from Client
- **Status**: 🟡 At Risk
- **Why**: `src/lib/services/translationService.ts` calls the MyMemory translation API directly (no auth, no proxying through your server). This means your users' flashcard text is sent from their browser to a third-party API. Additionally, there is no rate limiting, so a user could make unlimited translation requests burning your MyMemory quota.
- **Fix**: Proxy the translation call through a Next.js API route (`/api/translate`) that checks auth and enforces a per-user rate limiter (e.g., 20 requests/minute). This also hides the third-party API dependency from the client.

## 5. Infrastructure & Secrets

### 5.1 — Google Service Account Private Key Committed to Repository
- **Status**: 🔴 **CRITICAL — Act Immediately**
- **Why**: `credentials/TTS_service.json` contains a live RSA private key for a Google Cloud Service Account (`tts-bot@lingocoon-tts.iam.gserviceaccount.com`). This file is in the project directory. Although it is listed in `.gitignore`, the `.gitignore` entry only prevents future commits — if this was ever committed even once, the key is in your git history forever.

  A service account private key gives programmatic, permanent access to your Google Cloud project with the same permissions as that account. Unlike a password, rotating it doesn't expire old tokens immediately.
- **Attack scenario**: If this repo is ever pushed to GitHub (public or private with stolen credentials), an attacker extracts the private key and uses it to make unlimited Google Cloud TTS API calls, or any other action the service account is permitted to do on `lingocoon-tts`. This would result in financial charges to your Google Cloud billing account.
- **Fix** (two steps — do both):
  1. **Revoke the key NOW** in the Google Cloud Console → IAM → Service Accounts → `tts-bot` → Keys → Delete key `fdda8b4d...`. Creating a new key is fast; damage from a leaked key is not.
  2. **Move the credentials** out of the project directory entirely. Store them in an environment variable using the JSON content as a single-line string:
     ```env
     GOOGLE_APPLICATION_CREDENTIALS_JSON={"type":"service_account","private_key":"..."}
     ```
     Then parse in code: `JSON.parse(process.env.GOOGLE_APPLICATION_CREDENTIALS_JSON!)`.

### 5.2 — Google OAuth Client Secret in `.env.local`
- **Status**: 🟡 At Risk (depending on git history)
- **Why**: `.env.local` contains:
  ```env
  SUPABASE_AUTH_GOOGLE_CLIENT_SECRET=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
  ```
  The file itself is in `.gitignore` (`*.env*`), so it's safe from commits going forward. However:
  - If `.env.local` was ever mistakenly committed and then removed, the secret is in git history.
  - If someone gets read access to your Mac, they can read this file.
- **Fix**: Run `git log --all -- .env.local` to check if it was ever committed. If yes, rotate the secret in the Google Cloud Console → APIs & Services → Credentials. Use `git filter-repo` to purge it from history, then force-push.

### 5.3 — No Rate Limiting on Any Route
- **Status**: 🔴 At Risk
- **Why**: There is no rate limiting middleware, no IP-based throttling, and no per-user quota enforcement on any route — including the TTS synthesis endpoint. Since TTS calls trigger paid Google Cloud API usage, this is a financial risk.
- **Attack scenario**: A user (authenticated or not — see 3.1) writes a script that sends 1,000 requests/second to `/api/tts/synthesize`. Your Google Cloud bill spikes, possibly hitting spending limits and taking down TTS for all users.
- **Fix**: Use the `@upstash/ratelimit` package with Vercel KV (or any Redis-compatible store) to cap requests per user per minute on the TTS endpoints. Example:
  ```ts
  import { Ratelimit } from '@upstash/ratelimit';
  const ratelimit = new Ratelimit({ redis, limiter: Ratelimit.slidingWindow(10, '1m') });
  const { success } = await ratelimit.limit(user.id);
  if (!success) return NextResponse.json({ error: 'Rate limit exceeded' }, { status: 429 });
  ```

### 5.4 — Supabase Anon Key Exposure
- **Status**: ✅ Secure (by design)
- **Why**: `NEXT_PUBLIC_SUPABASE_ANON_KEY` is intentionally public. It identifies the project but doesn't grant admin access. Security is enforced by RLS policies on the database itself, not by keeping this key secret. The `service_role` key (which bypasses RLS) is correctly absent from both the `.env.local` and the codebase entirely.
- **Fix**: No action needed. Verify RLS is enabled (see 2.3).

### 5.5 — HTTPS Enforcement
- **Status**: 🟡 Unknown
- **Why**: The `next.config.ts` has no configuration at all (it's an empty config object). HTTPS enforcement depends entirely on your hosting platform (Vercel, etc.).
- **Fix**: If deploying on Vercel, HTTPS is automatic. If self-hosting, add HSTS headers via `next.config.ts`:
  ```ts
  headers: async () => [{
    source: '/(.*)',
    headers: [{ key: 'Strict-Transport-Security', value: 'max-age=63072000; includeSubDomains' }]
  }]
  ```
